# CTE — Code Triangulation Engineering

C++ 代码理解与重构的方法论框架。

不是规则清单。是一种思维方式：面对不透明的遗留代码，如何系统性地建立可靠认知，然后安全地改变它。

---

## 1. 核心原则

1. **单源不可信** — 任何单一信息源都有盲区。grep 不懂语义，AST 不跨文件，codegraph 不深入函数体。结论必须由 ≥2 个独立来源交叉确认。
2. **语义 > 文本** — C++ 的真相在语义层（类型、继承、模板、宏展开），不在文本层。文本匹配是起点不是终点。
3. **地图 > 领土** — 大文件先建地图再导航。直接读 50K tokens 的原始代码会丢失结构感，就像没有地图走进一座城市。
4. **量化 > 感觉** — "这个函数很大"不如"这个函数 847 行，圈复杂度 34，被 12 处调用"。工具给你数字，数字给你判断力。
5. **增量 > 全量** — 遗留项目有几千个既有警告。只看你引入的增量，否则信号被噪声淹没。

---

## 2. 信息源分类法

理解 C++ 代码有六个信息层级。每个层级看到不同的切面，也有不同的盲区。

| 层级 | 来源 | 看到什么 | 盲区 |
|------|------|---------|------|
| **L1 文本模式** | grep / rg / Bash | 字符串匹配、命名约定、注释线索 | 不懂类型、继承、宏展开、模板特化 |
| **L2 结构概览** | cpp-scan / cpp-bigfile-map / cpp-seam-finder / ctags | 文件级/函数级骨架、规模排行、段落划分 | 启发式，边界可能偏移，不理解语义 |
| **L3 语义精确** | clang_ast_list_functions / clang_ast_globals / clang_ast_virtual_calls / clang_ast_macro_jungle | 精确函数边界、圈复杂度、链接类型、虚函数分发、宏展开 | 依赖 compile_commands.json，单文件视角 |
| **L4 关系图谱** | codegraph callers / callees / impact / context | 跨文件调用链、符号影响范围、上下游依赖 | 索引可能过时，不深入函数体内部 |
| **L5 质量信号** | clang-tidy / cppcheck / cpp-quality-gate | 静态分析告警、代码异味、增量质量变化 | 既有告警噪声大，误报需人工过滤 |
| **L6 行为验证** | 编译 / 测试 / cpp-pipeline / cpp-characterize | 代码实际运行时的行为，重构前后的等价性 | 测试覆盖不全时无法证明等价 |

**关键认知**：层级越高越精确，但越依赖前置条件（compile_commands.json、.codegraph 索引、测试套件）。高层不可用时降级到低层，但必须标注置信度。

## 3. 诊断流程：Question → Sources → Triangulate → Act

### Step 1 · 明确问题类型

面对 C++ 代码，先问自己：**我想知道什么？**

所有问题归入五类：

| 类型 | 典型问题 |
|------|---------|
| **边界** | 这个函数从哪到哪？多大？多复杂？ |
| **影响** | 改这个符号会影响谁？爆炸半径多大？ |
| **依赖** | 这个函数/类依赖什么全局状态？被谁调用？调用谁？ |
| **风险** | 这里有哪些接缝、宏丛林、裸指针、危险转换？ |
| **验证** | 我的改动安全吗？行为没变吧？ |

### Step 2 · 选择信息源组合

每类问题有推荐的源组合。**至少选 2 个层级**。

| 问题类型 | 首选组合 | 降级方案（前置条件不满足时） |
|---------|---------|--------------------------|
| **边界** | L3 (AST 精确行号 + 圈复杂度) + L2 (段落上下文) | L2 (ctags 边界) + L1 (grep 辅助) |
| **影响** | L4 (codegraph impact) + L3 (虚函数调用点) | L4 + L1 (grep 调用名，标注"可能漏虚函数") |
| **依赖** | L4 (callers/callees) + L3 (globals) + L2 (include 图) | L1 (grep) + L2 (seam-finder)，标注"启发式" |
| **风险** | L2 (seam-finder 9 类扫描) + L3 (AST 精确分类) + L4 (impact) | L2 单独使用，标注"30% 误报率" |
| **验证** | L6 (编译+测试+静态分析) + L5 (增量质量门禁) | L6 仅编译，标注"无测试覆盖" |

### Step 3 · 交叉验证

拿到多个来源的结果后：

- **一致** → 置信度高，可以行动
- **不一致** → 信任层级高的来源（L6 > L4 > L3 > L2 > L1），或用第三个来源仲裁
- **某来源返回空** → 不一定是"没有"，可能是前置条件不满足（索引未建、compile_commands 缺失）。先检查前置条件

### Step 4 · 行动并验证

每次代码变更后，必须回到 L6（行为验证）确认安全。不验证的变更不算完成。

## 4. 决策表 —— 场景到源组合

| 场景 | 错误做法 | 正确做法 |
|------|---------|---------|
| 拿到一个陌生项目 | grep 几个关键词就开始改 | L2 (cpp-scan) 建全局认知 → L4 (codegraph status) 看索引状态 → 再深入 |
| 面对 5000 行文件 | Read 全文，试图理解 | L2 (cpp-bigfile-map) 建地图 → 只 Read 需要的段落 → L4 (callers/callees) 看关系 |
| 要抽取一个函数 | 直接找大括号边界开始切 | L3 (AST 精确边界) + L4 (callers 确认调用方) + L2 (段落上下文确认切分点) |
| 要改一个全局变量 | grep 所有引用就改 | L3 (globals 分类链接类型) + L4 (impact 看影响范围) + L6 (characterize 锁定行为) |
| 要清理 #ifdef | 读代码理解每个分支 | L3 (macro_jungle 量化复杂度) + L2 (seam-finder 定位) + L1 (clang -E 看展开) |
| 要评估一个重构方案 | 凭经验判断风险 | L4 (impact 量化受影响符号数) + L3 (虚调用点) + L5 (baseline 记录当前质量) |
| 改完代码 | 看一眼没报错就提交 | L6 (pipeline 编译+测试+静态分析) + L5 (quality-gate check 看增量) |

## 5. 信任层级 —— 来源冲突时该信谁

当不同来源给出矛盾结论时：

```
L6 行为验证（测试通过/失败）
  ↑ 最高信任
L4 关系图谱（codegraph 调用链）
  ↑
L3 语义精确（AST 分析）
  ↑
L2 结构概览（启发式扫描）
  ↑
L1 文本模式（grep）
  ↑ 最低信任
```

**原则**：高层来源否定低层结论时，信高层。低层来源否定高层结论时，怀疑低层（通常是 grep 漏掉了语义信息）。

---

## 6. 置信度标注

每次判断都应附带置信度。不标注等于假装确定。

| 置信度 | 条件 | 行动 |
|--------|------|------|
| **高** (≥80%) | ≥2 个高层来源（L3+）一致 | 可以行动 |
| **中** (50-80%) | 仅低层来源，或来源间有分歧 | 行动但标注风险，增加验证 |
| **低** (<50%) | 仅 L1/L2，或前置条件不满足 | 不行动，先获取更多信息 |

---

## 7. 降级策略

高层工具不可用时，不是停下来，而是降级并标注。

| 缺失条件 | 降级路径 | 标注 |
|---------|---------|------|
| 无 compile_commands.json | L3 → L2（正则启发式） | `[降级: 正则 ~30% 误报]` |
| 无 .codegraph 索引 | L4 → L1（grep）+ L2（seam-finder） | `[降级: grep 会漏虚函数]` |
| 无测试套件 | L6 → 仅编译验证 | `[降级: 无行为等价性证明]` |
| 无 clang-tidy | L5 → 仅 cppcheck + 编译 | `[降级: 静态分析不完整]` |

**关键**：降级不是放弃，是带着已知的不确定性继续。标注让后续的人（或下一个 session 的你）知道这里的判断有多可靠。

## 8. 反模式清单

| 反模式 | 为什么错 | 正确做法 |
|--------|---------|---------|
| 一个 grep 就下结论 | grep 不懂 C++ 语义，会漏掉 80% 的真实情况 | 至少 2 个层级交叉验证 |
| Read 整个大文件 | 50K tokens 超出注意力窗口，丢失结构 | 先建地图（bigfile-map），再导航到段落 |
| 凭经验判断影响范围 | 人会漏掉间接调用、虚函数分发、函数指针 | codegraph impact 量化 |
| 改完不验证 | "看起来没问题"不是证据 | pipeline + quality-gate 提供证据 |
| 全量看告警 | 几千个既有告警淹没你引入的新问题 | quality-gate baseline + check 只看增量 |
| 高层不可用就停下来 | 降级不是失败，是带标注的继续 | 降级 + 标注置信度 |
| 不标注置信度 | 假装确定比承认不确定更危险 | 每次判断附带高/中/低 |
| 同一来源验证两次 | 不是交叉验证，是自我确认 | 必须跨层级 |

## 9. 记忆面

CTE 的判断质量取决于积累的项目知识。六个记忆面：

| 面 | 读时机 | 写时机 |
|----|--------|--------|
| **项目状态** (REFACTOR_STATE / PIPELINE_STATE) | session 开始 | 每个阶段完成 |
| **质量基线** (QUALITY_BASELINE) | 变更前 | 首次分析时记录 |
| **工具缺口** (TOOL_GAPS) | 规划时 | 工具失败时 |
| **分区台账** (PARTITION_LEDGER) | 选择下一个任务时 | 分区完成时 |
| **持久记忆** (Hindsight / 记忆插件) | 任务开始查询相关经验 | 发现坑、学到教训时 |
| **代码库本身** (git history / blame) | 调试前 | 提交时写精确 message |

---

## 10. 纠错规则

- **同一问题三次判断错误** → 停，质疑信息源组合是否有系统性盲区，换组合
- **工具返回空结果** → 不假设"没有"，先检查前置条件（索引存在？compile_commands 包含此文件？）
- **来源间矛盾** → 不选边站，引入第三个来源仲裁
- **降级后判断被推翻** → 记录这个降级失败案例，下次同类场景优先获取高层来源

## 11. 与 AHE 的对应关系

| AHE 概念 | CTE 对应 |
|---------|---------|
| 删行为指导，留硬约束 | 删"你应该用 X 工具"，留"单源不可信"（原则） |
| 组件分类法（16 层） | 信息源分类法（6 层） |
| evaluate → analyze → improve | Question → Sources → Triangulate → Act |
| 决策表（症状→层级） | 决策表（场景→源组合） |
| 可证伪预测 | 置信度标注 + 降级标注 |
| 反模式清单 | 反模式清单 |
| 记忆面（6 个） | 记忆面（6 个） |
| 纠错规则 | 纠错规则 |

AHE 优化的是**提示词系统**——让 AI 写出更好的代码。
CTE 优化的是**认知过程**——让 AI 更可靠地理解不透明的遗留代码。

两者互补：AHE 管"怎么写"，CTE 管"怎么懂"。
